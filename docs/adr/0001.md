# ADR-001: Define Communication Roles and Boundaries

## Status

Proposed

## Context

The system consists of components and clients. A component may run as a separate service/process or as an in-process module inside a monolith. Components need to exchange commands, status updates, logs, presence information, and high-rate data.

A single communication mechanism is not appropriate for all of these needs. A command, a status update, a log message, a state snapshot, and an image stream have different semantics.

Before selecting a concrete transport or deployment style, the system must define the communication roles and their boundaries.

## Decision

The system communication model is divided into six roles:

### 1. Control

Control is used for explicit asynchronous commands.

A command expresses intent to perform an action. Domain state is not updated from a direct command reply. Results and state changes are observed through events and state snapshots.

Examples:

- set parameter
- start job
- stop job
- load configuration
- request one measurement
- query device information

Control communication has:

- command id
- target component
- command type / method name
- parameters
- sender identity
- timestamp
- optional correlation id

Control is not used for high-rate streams or continuous status updates.

### 2. Events

Events are immutable facts that something happened.

Examples:

- component_started
- parameter changed
- job started
- job finished
- alarm raised
- alarm cleared
- defect detected

Events are used for online updates and component-to-component notification.

Events are not the source of current truth for late subscribers.

### 3. State

State represents the current truth exposed by a component.

Examples:

- current component status
- current parameters
- active job
- current device connection state
- latest known measurement summary

State is used by clients that start late, reconnect, or detect that they missed events.

### 4. Logs

Logs are human-oriented diagnostic and operator messages.

Examples:

- requested exposure was clamped
- device reconnect failed
- calibration file could not be loaded
- stream frame was dropped

Logs are separate from events. Events are domain facts; logs are diagnostics.

Logs must support future translation through stable message ids and runtime arguments.

### 5. Streams

Streams are high-rate or large-volume data channels.

Examples:

- images
- point clouds
- robot poses
- coordinate arrays
- measurement samples

Streams are not transported as control responses, events, or logs.

Streams have their own backpressure, dropping, encoding, and binary layout rules.

### 6. Presence

Presence is used to report runtime liveness of configured components.

Examples:

- component is online
- component type/name/instance
- component version
- contract version or contract hash
- lifecycle status

Presence allows clients and other components to know which configured components are online. It is not open discovery and does not define new runtime capabilities.

## Consequences

The system avoids forcing all communication through one abstraction.

Each communication role can have an appropriate contract, payload shape, and delivery expectation.

Transport selection is deferred to a later ADR.

The same communication model should be implementable over different transports.

## Non-decisions

This ADR does not select a transport.

This ADR does not define topic names, key syntax, URLs, or broker-specific routing.

This ADR does not define exact schemas for every component type.

This ADR does not require a monorepo, central runtime service, or distributed deployment.
